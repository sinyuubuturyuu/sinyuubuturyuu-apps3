# Firebase Authentication 仕様書

## 目的

`dev` 環境で、メールアドレス + パスワードによるログイン機能を導入する。

今回の目的は、未ログインではアプリを使えない状態にし、今後の社員特定連携と Firestore Security Rules 本番化の前提を作ること。

## 対象環境

- 対象: `dev` Firebase プロジェクト
- 認証方式: Firebase Authentication の `メール / パスワード`
- 対象アプリ:
  - `dev/sinyuubuturyuu/index.html`
  - `dev/sinyuubuturyuu/getujitiretenkenhyou/`
  - `dev/sinyuubuturyuu/getujinitijyoutenkenhyou/`

## 今回の対象範囲

今回実装するのは「認証導入の第1段階」とする。

### 実装するもの

- ログイン画面
- ログイン状態の監視
- 未ログイン時の画面ガード
- ログアウト
- 認証エラー表示
- 認証状態の維持確認

### 今回はまだ実装しないもの

- 社員マスタとの対応付け
- 管理者 / 一般利用者の権限分離
- パスワード再発行
- 招待フロー
- 本番 Firebase への反映
- Firestore Security Rules の本番最終化

## 画面方針

### 1. 初期表示

`dev/sinyuubuturyuu/index.html` を開いたとき、

- 未ログインならログイン画面を表示する
- ログイン済みなら従来のランチャー画面を表示する

### 2. ログイン画面

表示項目は次の最小構成とする。

- メールアドレス入力欄
- パスワード入力欄
- ログインボタン
- エラーメッセージ表示欄

必要に応じて以下を追加してよい。

- パスワード表示 / 非表示切替
- 読み込み中表示

### 3. ランチャー画面

ログイン後は従来の `シンユウ物流アプリ` ランチャーを表示する。

追加する要素:

- ログアウトボタン
- ログイン中ユーザーの簡易表示
  - 例: メールアドレス

### 4. 各アプリ画面

対象アプリを URL 直打ちで開いた場合も、

- 未ログインならログイン画面へ戻す、またはランチャーへ戻す
- ログイン済みなら通常どおり表示する

## 認証仕様

### 使用 API

- `onAuthStateChanged`
- `signInWithEmailAndPassword`
- `signOut`

必要に応じて初期化用に以下を使う。

- `initializeApp`
- `getAuth`

### ログイン成功時

- Firebase Auth のログイン状態を取得する
- ランチャー画面を表示する
- 既存の設定データ表示はそのまま維持する

### ログイン失敗時

ユーザーに分かる短い文言を表示する。

例:

- メールアドレスまたはパスワードが違います
- 入力内容を確認してください
- 一時的にログインできません

内部の Firebase エラーコードは画面へそのまま出さない。

### ログアウト時

- Firebase Auth からサインアウトする
- ログイン画面へ戻す
- アプリ本体は表示しない

### ログイン状態の維持

ブラウザ再読込後も Firebase Auth のセッションによりログイン状態を維持する。

ただし、明示的にログアウトした場合は未ログイン状態へ戻る。

## 画面遷移仕様

### A. 未ログインでランチャーへアクセス

1. Auth 初期化
2. ログイン状態確認
3. ログイン画面表示

### B. ログイン成功

1. メール / パスワードでサインイン
2. `onAuthStateChanged` でログイン済みを検知
3. ランチャー画面表示

### C. ログイン済みでアプリへアクセス

1. Auth 初期化
2. ログイン状態確認
3. 対象アプリ表示

### D. 未ログインでアプリ直URLへアクセス

1. Auth 初期化
2. 未ログインを検知
3. ランチャーのログイン画面へ戻す、または未ログイン画面を表示

## 実装方針

### 1. 認証処理の共通化

`dev/sinyuubuturyuu/` 配下に認証用の共通 JS を作成し、ランチャーと各アプリで共通利用する。

想定役割:

- Firebase Auth 初期化
- 現在ユーザー取得
- ログイン / ログアウト
- 認証状態変更監視
- 未ログイン時のガード

### 2. ログイン UI の分離

ランチャー内に直接埋め込むか、共通のログインコンポーネント相当の DOM を作る。

初期段階では、複雑な画面分離よりも

- HTML 内にログイン領域
- HTML 内にアプリ領域
- 認証状態で表示切替

を優先する。

### 3. Firestore 接続前の認証確認

既存の Firebase 連携処理は、可能な範囲で

- 認証完了後に起動する
- 未ログインなら Firestore を使う処理を走らせない

ようにする。

### 4. 匿名認証の扱い

今回の `dev` 実装が安定したら、関連アプリの `useAnonymousAuth: true` を段階的に廃止する前提とする。

ただし、今回の仕様書段階では「置き換え前提を決める」まででよい。

## 受け入れ条件

### 基本

- 未ログインではランチャーを使えない
- 正しいメール / パスワードでログインできる
- ログイン後にランチャーが表示される
- ログアウトできる
- ログアウト後は再びログイン画面に戻る

### 画面遷移

- `dev/sinyuubuturyuu/index.html` 直アクセスで動作する
- `月次タイヤ点検表` 直アクセスでも未ログインなら使えない
- `月次日常点検表` 直アクセスでも未ログインなら使えない

### エラー

- パスワード誤り時に適切なエラーが出る
- ネットワーク不良時に固まらない
- ログイン中の処理中表示がある

### 維持

- 再読込後もログイン状態が維持される
- サービスワーカーやキャッシュの影響で古いログイン画面が残りにくい

## 実装順

1. 共通認証モジュールを作る
2. ランチャーにログイン画面を追加する
3. ログイン / ログアウトを実装する
4. ランチャーの未ログインガードを入れる
5. 月次タイヤ点検表へ未ログインガードを入れる
6. 月次日常点検表へ未ログインガードを入れる
7. エラー表示と再読込確認を行う

この仕様書の完了条件は次のとおり。

- `dev` でメール / パスワード認証の対象範囲が明確になっている
- 画面、遷移、受け入れ条件が決まっている
- 実装順が決まっている

このあと、仕様書に沿って `dev` 実装へ着手する。
